项目实施

工程实现逻辑通常指的是将一个项目从概念转化为实际可交付产品或服务的过程。这个过程涉及多个阶段，每个阶段都有特定的目标和任务。以下是一个典型的软件工程项目实现逻辑：

  • 需求分析：
    • 收集和分析用户需求。
    • 制定详细的功能规格说明书。
    • 确认所有利益相关者对需求的一致理解。
  • 系统设计：
    • 构建系统架构，包括高层次设计（如模块划分）和详细设计（如数据库结构、接口定义）。
    • 选择合适的技术栈。
  • 实施与编码：
    • 根据设计文档编写代码。
    • 执行单元测试以验证每个模块功能是否正确。
  • 集成与测试：
    • 集成各个模块，进行系统级测试。
    • 发现并修复缺陷，确保整体系统按预期工作。
  • 部署：
    • 将软件发布到生产环境中。
    • 配置必要的硬件和软件环境，确保其稳定运行。
  • 维护与支持：
    • 持续监控系统性能，并根据反馈进行优化和更新。
    • 修复可能出现的问题或漏洞，提供技术支持。
  • 评估与改进：
    • 回顾整个开发过程，总结经验教训。
    • 在未来项目中应用所学知识，以提高效率和质量。

在实际操作中，这些步骤可能会重叠或迭代进行。例如，在敏捷开发方法论下，各个阶段可能会更频繁地重复，以便快速响应变化。在大型复杂项目中，还需要考虑风险管理、资源配置、时间计划等因素。这一系列活动共同保障了工程实现逻辑能够顺利推进，实现预期目标。对于项目完成到某一阶段，需要添加新需求时，按照下面方法进行落实：

  • 方案
    • 该需求实现有哪些方案？
    • 每个方案的优缺点？确定最佳方案
  • 步骤
    • 实现上述已确定的方案需要哪些步骤？
    • 每个步骤对已有功能的影响是什么？
  • 实施
    • 根据上述步骤逐步进行
  • 核实验证
    • 如何确定新需求已实现
    • 其他已有功能是否有异常

通用bug汇总

  • xxx is null引起的报错
    • 原因：数据加载完前xxx变量被执行
    • 解决：通过条件语句判断，在所有数据加载完后再执行后面的语句
  • 列表页标签切换问题
    • 场景：当前页面数据获取了50条数据后（offset为50）点击切换标签，导致下一页获取数据后offset还是为50
    • 解决：重置为第一页刷新(offset为0，加载列表)

登录/注册

通过条件语句判断view类型，code表示验证码登录，signup表示注册，login表示用户名密码登录

  • 平台登录注册接口
    • 用户名密码登录：Dapi的用户登录注册接口 -> 用户名密码登录
    • 手机号验证码登录：Dapi的用户登录注册接口 -> 手机号验证码登录
    • 注册：Dapi的用户登录注册接口 -> 手机号验证码注册接口
    • 小程序手机号授权登录：Dapi的用户登录注册接口 -> 使用微信授权获取的手机号登录
  • 登录流程： 同意协议条款 -> 判断表单内容是否为空 -> 登录接口 -> 绑定密码的状态 变量为空 -> 使用token登录
  • 微信小程序：获取用户信息，如果没有用户信息切换为手机号登录注册页
  • 用户token登录：
    • 接口：小程序 -> 使用用户token在移动端登录（小程序/rnapp）(dw.setSiteuserToken)

特殊场景

  • 一个用户用一个手机号注册两个用户，身份不同，比如保险销售员入职+离职前后作为客户，平台账号如何区分？
    • 解决方案：给员工手机号前面加上“+”作为区分，其实是两个uer，删除一个对另外一个不影响，但需要注意发送短信时可能显示验证码不匹配

金额误差问题
  • 先四舍五入后加、先加后四舍五入两种计算方式会存在精度误差
    • 先四舍五入后加：100.05 + 200.05 = 300.10 --> 100.10 + 200.10 = 300.20
    • 先加后四舍五入：100.05 + 200.05 = 300.10 --> 100.05 + 200.05 = 300.10

字符编码和加密解密

为什么用base64编码和hex编码传递参数
  • 相比utf-8和其他编码，base64和hex编码更加准确不会出现乱码问题，所以一般加密数据时用base64编码或hex编码
  • 加密解密工具：
    • CryptoJS：crypto-js主要专注于对称加密（如 AES、DES）、哈希算法（如 MD5、SHA 等）和流式加密（如 RC4、Rabbit 等）等方面的支持。
    • JSEncrypt：非对称加密，非对称加密算法中常用的就是 RSA

四、常见性能优化方法

0. 性能问题调试方法

  • 0.1 查看网络请求耗时，找出耗时的请求。开发者工具选择 Network，过滤 XHR 请求，可以根据 Time 列分析请求耗时。 一般 <100ms 属于比较好的情况，100-200ms 可以接受，再往上就要考虑能不能优化了。
  • 0.2 小程序如果修改状态后卡顿可能是状态数据量太大
  • 0.3 可以使用手机网页小程序调试小程序性能问题

1. 控制器常见性能问题

  • 避免在循环中大量查询数据
  • 合理使用批量创建接口，避免在循环中创建数据
  • 大量数据操作使用事件异步处理
  • 使用并发执行合并请求
  • 区分主次请求，控制页面加载时序，复杂的页面可能需要很多查询，但是我们可以在主要查询结束之后就显示出页面，其他的可以慢慢加载。 比如详情页，最重要查询是对应详情，其他如评论收藏等信息可以放到后面加载
  • 减少关联查询数量和深度
  • 减少外键过滤数量
  • 尽量避免深度拷贝数据，尤其是大数据量的拷贝，可以使用浅拷贝
  • 通过使用索引修改部分数据优化性能
  • 使用组合功能遍历数据库

2. 小程序性能优化

  • 通过选择部分字段减少状态数据量
  • 通过数据映射减少状态数据量
  • 合并相邻修改状态动作
  • 通过全局变量减少状态数据量
  • 图片需要开启裁剪并合理设置图片尺寸：列表页小图 200200详情页大图 500500
  • 滑动容器避免监听滚动事件
  • 使用文本组件而不是原子组件
  • 尽可能用少的组件实现功能，勾选不要的组件可以直接删除释放空间

3. PC 端使用预加载优化

  • 3.1 通过事件处理详情页 TDK
  • 3.2 通过搜索引擎提高网页性能
