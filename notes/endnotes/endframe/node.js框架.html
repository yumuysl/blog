Express.js

项目创建

  • 全局安装express-generator：pnpm i express-generator -g
    • express-generator是一个创建express项目的脚手架工具，不用一个一个创建项目文件，适合需要高度定制化的场景
      • 注：ts版本的，直接下载express + ts库及其他库，目前cli工具都不好用
  • 创建项目：express 项目名，如果不需要视图express --no-view 项目名
  • 安装依赖：pnpm i
  • 启动项目：pnpm start，在浏览器打开localhost:3000
  • 项目结构：
    • bin：存放可执行文件
    • public：存放静态文件
    • routes：存放路由文件

启动项目-第一个接口

  • 删除public文件夹下面的index.html文件
  • 在routes文件夹下的index.js文件中，将res.render('index', { title: 'Express' });改为res.json({ "message": 'hellExpress' });
  • 重启服务，刷新浏览器，如果看到{"message":"hellExpress"}，则说明接口创建成功
  • 问题：每次修改代码都需要重启服务，怎么自动重启服务？
    • 安装nodemon：pnpm i nodemon -D
    • 在package.json文件中，将"start": "node ./bin/www"改为"start": "nodemon ./bin/www"
    • 重启服务，修改代码后，nodemon会监听到修改，自动重启服务

RESTfull API规范

核心原则

  1. 资源导向
    • URI 表示资源（名词，非动词），如 /users、/articles/{id}/comments
    • 使用复数形式（如 /users 而非 /user）
  2. HTTP 方法语义化
    • GET：获取资源
    • POST：创建资源
    • PUT：全量更新资源
    • PATCH：部分更新资源
    • DELETE：删除资源
  3. 无状态性
    • 每个请求需包含完整上下文，服务端不保存客户端会话状态。

url设计规范

  1. 层级清晰
    • 多层级资源用 / 分隔，如 /users/{userId}/orders
  2. 过滤、分页、排序
    • 使用查询参数：?status=active&page=2&limit=10&sort=-createdAt
  3. 版本控制
    • URI 路径：/api/v1/users
    • 请求头：Accept: application/vnd.myapi.v1+json

HTTP 状态码

状态码含义典型场景
200OK成功响应（GET/PUT/PATCH）
201Created资源创建成功（POST）
204No Content成功但无返回体（DELETE）
400Bad Request请求参数错误
401Unauthorized未认证
403Forbidden无权限访问
404Not Found资源不存在
500Internal Server Error服务端内部错误

请求与响应格式

  1. 请求头
    • Content-Type: application/json
    • Accept: application/json
  2. 响应体
    • 统一返回 JSON 格式，包含明确的数据结构：
    	 {
		 "data": { ... },  // 主数据
		 "meta": {         // 分页/元信息
			 "page": 1,
			 "total": 100
		 },
		 "error": {        // 错误信息（可选）
			 "code": "INVALID_PARAM",
			 "message": "参数校验失败"
		 }
	 }

安全性与认证

  1. HTTPS
    • 强制使用 HTTPS 加密通信。
  2. 认证机制  、
    • OAuth 2.0 / JWT 令牌认证（通过 Authorization 头传递）。
  3. 权限控制
    • RBAC（基于角色的访问控制）限制资源操作权限。

错误处理

  1. 统一错误格式
{
 "error": {
	 "message": "用户不存在",
	 "code": "RESOURCE_NOT_FOUND",
	 "details": "用户 ID 123 未找到"
 }
}
  1. 避免返回裸错误信息，防止泄露敏感数据。

HATEOAS（超媒体驱动）

  • 在响应中返回资源关联的链接，实现 API 自描述性：
{
	"id": 123,
	"name": "Alice",
	"_links": {
		"self": { "href": "/users/123" },
		"orders": { "href": "/users/123/orders" }
	}
}

设计建议

  1. 文档化
    • 使用 OpenAPI (Swagger) 生成交互式文档。
  2. 兼容性
    • 通过版本号管理不兼容的变更，避免破坏客户端。
  3. 性能优化
    • 支持 ETag 缓存、gzip 压缩、CORS 跨域配置。

结合ORM工具操作数据库

  • 预备：
    • 数据库使用mysql
    • 使用Prisma操作数据库(官网：https://prisma.org.cn/docs/orm/overview/introduction)
  • 安装prisma：pnpm install prisma --save-dev ，pnpm install @prisma/client
  • 初始化prisma：pnpm prisma init --output '../generated/prisma'
  • Prisma v7有破坏性更新，具体查看链接
    • 支持ESM，但与nestjjs结合时，依然采用CJM，在nestjs中采用依赖注入的方式使用prsima，查看官方案例
  • 连接数据库：
    • 更新.env文件的地址：DATABASE_URL="mysql://username:password@localhost:3306/your_database_name"
    • schema.prisma文件中，数据源db 的provider赋值为mysql
  • 定义模型
    • 如果数据库中没有表，在schema.prisma文件中通过模型定义，之后迁移prisma migrate dev --name init或pnpm prisma migrate dev --name add_new_models，注意该命令会自动pnpm prisma generate更新客户端
    • 如果数据库中有数据，执行内省pnpm prisma db pull语句，注意内省后需要pnpm prisma generate更新客户端
    • 如果手动修改数据库字段，在实践中不推荐，会导致架构漂移，如果需要处理这样的情况，优先使用db pull更新schema.prisma文件，之后务必npx prisma migrate dev --name manual_changes创建一个新的迁移追认该变更；次选方案是标记为已解决，使用npx prisma migrate resolve --applied "20230901082301_init"
    • 手动修改schema模型，这是官方推荐的方式，运行npx prisma migrate dev --name add_new_field，但需要注意的是字段修改或者删除则数据库中该字段数据也会没有
      • 生产环境中：使用--create-only标志先生成迁移文件而不执行，然后手动编辑生成的 SQL，将其改为数据安全的操作（例如先创建新列、迁移数据、再删除旧列），最后再npx prisma migrate dev应用迁移；​严格使用 prisma migrate deploy​ 来应用已生成的迁移文件，绝不使用 db push或 migrate dev
      • 开发环境中：更自由地使用 migrate reset来重置数据库
场景核心问题推荐操作关键命令安全注意点
①数据库无数据，Schema 有模型需要根据模型创建数据库结构使用 migrate dev创建并应用迁移npx prisma migrate dev --name init安全
② 数据库有数据，Schema 无模型需要将现有数据库结构导入 Prisma使用 db pull自省数据库npx prisma db pull安全
③ 手动修改数据库字段出现“架构漂移”，历史记录与实际情况不符首选：通过 db pull同步并创建新迁移；次选：标记迁移状态npx prisma db pullnpx prisma migrate resolve ...​谨慎操作，避免重置导致数据丢失
④ 手动修改 Schema 模型需要将模型变更同步到数据库使用 migrate dev创建并应用新迁移npx prisma migrate dev --name update_model若涉及修改/删除字段，​需谨慎处理以防数据丢失

用户登录校验

密码加密和校验

  • 用户密码需要加密后再写入数据库
  • 使用第三方工具bcrypt.js处理加密解密，因为node.js自带crypto模块加密后的密码容易被破解
  • 加密和校验操作
    • 使用哈希加盐给密码加密：bcrypt.hashSync(password, 10)
    • 校验用户密码是否正确：bcrypt.compareSync(db_password, input_password)

中间件

中间件函数能够访问请求对象(req)、响应对象 (res) 以及应用程序的请求/响应循环中的下一个中间件函数。下一个中间件函数通常由名为 next 的变量来表示。

  • 中间件函数可以执行以下任务：
    • 执行任何代码
    • 对请求和响应对象进行更改
    • 结束请求/响应循环
    • 调用堆栈中的下一个中间件函数
    • 如果当前中间件函数没有结束请求/响应循环，那么它必须调用 next()，以将控制权传递给下一个中间件函数。否则，请求将保持挂起状态

分类

  • 应用层中间件：使用 app.use() 和 app.METHOD() 函数将应用层中间件绑定到应用程序对象的实例
app.use('/user/:id', function (req, res, next) {
  console.log('Request Type:', req.method);
  next();
});
  • 路由器层中间件：绑定到 express.Router() 的实例，使用 router.use() 和 router.METHOD() 函数装入路由器层中间件
router.use('/user/:id', function(req, res, next) {
  console.log('Request URL:', req.originalUrl);
  next();
}, function (req, res, next) {
  console.log('Request Type:', req.method);
  next();
});
  • 错误处理中间件：误处理函数有四个自变量而不是三个，专门具有特征符 (err, req, res, next)
app.use(function(err, req, res, next) {
  console.error(err.stack);
  res.status(500).send('Something broke!');
});
  • 内置中间件：从4.x起，以单独模块的形式提供，查看中间件列表
  • 第三方中间件：安装具有所需功能的 Node.js 模块，然后在应用层或路由器层的应用程序中将其加装入
//npm install cookie-parser
var express = require('express');
var app = express();
var cookieParser = require('cookie-parser');

// load the cookie-parsing middleware
app.use(cookieParser());

import { Controller, Get, Query, Post, Body, Put, Param, Delete } from '@nestjs/common';
import { CreateCatDto, UpdateCatDto, ListAllEntities } from './dto';

@Controller('cats')
export class CatsController {
 @Post()
 create(@Body() createCatDto: CreateCatDto) {
   return 'This action adds a new cat';
 }

 @Get()
 findAll(@Query() query: ListAllEntities) {
   return `This action returns all cats (limit: ${query.limit} items)`;
 }

 @Get(':id')
 findOne(@Param('id') id: string) {
   return `This action returns a #${id} cat`;
 }

 @Put(':id')
 update(@Param('id') id: string, @Body() updateCatDto: UpdateCatDto) {
   return `This action updates a #${id} cat`;
 }

 @Delete(':id')
 remove(@Param('id') id: string) {
   return `This action removes a #${id} cat`;
 }
}

提供者

  • 用 CLI 创建服务，只需执行 $ nest g service cats 命令
  • @Injectable() 装饰器将元数据附加到类上，表明 CatsService 是一个可以由 NestIoC 容器管理的类
  • 自定义提供者

import { Injectable, Optional, Inject } from '@nestjs/common';

@Injectable()
export class HttpService<T> {
  constructor(@Optional() @Inject('HTTP_OPTIONS') private httpClient: T) {}
}
  • 注册提供者
    • 通过编辑模块文件（app.module.ts）并将服务添加到 @Module() 装饰器的 providers 数组来实现
//app.module.ts
import { Module } from '@nestjs/common';
import { CatsController } from './cats/cats.controller';
import { CatsService } from './cats/cats.service';

@Module({
 controllers: [CatsController],
 providers: [CatsService],
})
export class AppModule {}

模块

  • 模块是一个用 @Module() 装饰器注解的类
  • @Module() 装饰器接收一个带有描述模块属性的对象
    • providers：将由 Nest 注入器实例化,且至少可在本模块内共享的提供者
    • controllers：本模块中定义的需要实例化的控制器集合
    • imports：导入模块的列表，这些模块导出了本模块所需的提供者
    • exports：本模块提供的 providers 子集，这些提供者应可供导入本模块的其他模块使用。可以使用提供者本身或其令牌（provide 值）
  • 使用 CLI 创建模块，只需执行 $ nest g module cats 命令

// app.modules.ts
import { Module, NestModule, RequestMethod, MiddlewareConsumer } from '@nestjs/common';
import { LoggerMiddleware } from './common/middleware/logger.middleware';
import { CatsModule } from './cats/cats.module';

@Module({
 imports: [CatsModule],
})
//模块类的 configure() 方法，包含中间件的模块必须实现 NestModule 接口
export class AppModule implements NestModule {
 configure(consumer: MiddlewareConsumer) {
   consumer
     .apply(LoggerMiddleware)
     .forRoutes({ path: 'cats', method: RequestMethod.GET });
 }
}

异常过滤器

  • 补充
    • 由内置的全局异常过滤器实现，它能处理 HttpException 类型（及其子类）的异常。当遇到无法识别的异常（既不是 HttpException 也不是其继承类）时，内置异常过滤器会生成以下默认 JSON 响应
    {
  "statusCode": 500,
  "message": "Internal server error"
}
    • 内置了很多继承自基础 HttpException 标准异常的常见HTTP异常：
      • BadRequestException、UnauthorizedException、ForbiddenException、RequestTimeoutException、HttpVersionNotSupportedException、GoneException
      • 案例
      throw new BadRequestException('Something bad happened', {
  cause: new Error(),
  description: 'Some error description',
});
  • 整体案例
@Get()
async findAll() {
 try {
   await this.service.findAll()
 } catch (error) {
   throw new HttpException({
     status: HttpStatus.FORBIDDEN,
     error: 'This is a custom message',
   }, HttpStatus.FORBIDDEN, {
     cause: error
   });
 }
}
