前端工程化梳理

禹穆穆2025/3/20...大约 6 分钟前端笔记

总览

代码规范体系

  1. Linter：ESLint + Prettier 保证代码风格统一（可配置 Airbnb/Standard 等流行规范）。
  2. Git 提交规范：commitlint + Husky 约束提交信息格式（如 Angular Commit Message 规范）。
  3. 类型检查：TypeScript 静态类型系统提升代码健壮性。

模块化与架构

  1. 组件化开发：使用 Storybook 搭建 UI 组件库，实现文档与开发同步。
  2. 状态管理：复杂场景引入 Redux/MobX/Zustand 管理全局状态。
  3. 路由管理：React Router 实现 SPA 路由控制，结合懒加载优化性能。

自动化测试

  1. 单元测试：Jest + React Testing Library 测试组件逻辑。
  2. E2E 测试：Cypress/Playwright 模拟用户行为测试核心流程。
  3. 覆盖率报告：Istanbul 生成测试覆盖率报告，保障代码质量。

构建与部署优化

  1. 打包优化：Vite 内置 Rollup 配置拆分 chunk，结合 Tree Shaking 剔除无用代码。
  2. CDN 加速：静态资源上传至 CDN，使用 hash 文件名强缓存。
  3. CI/CD 流水线：GitHub Actions/Jenkins 实现自动化构建、测试、部署。

监控与性能

  1. 性能分析：Lighthouse/Web Vitals 监控首屏加载、交互延迟等指标。
  2. 错误追踪：集成 Sentry 实时捕获生产环境异常。
  3. 日志管理：前端日志上报至 ELK（Elasticsearch+Logstash+Kibana）分析。

辅助工具链

  1. Mock 数据：Mock.js 或 MSW 拦截 API 请求，实现前后端并行开发。
  2. 文档生成：Docusaurus/VitePress 自动生成项目文档。
  3. 依赖管理：使用 pnpm 替代 npm/yarn，利用硬链接节省磁盘空间。

🔍 扩展建议：

  • 微前端架构（qiankun/Module Federation）适用于大型项目拆分。
  • 使用 TurboRepo 管理多包仓库，解决 Monorepo 协作问题。
  • 容器化部署（Docker + K8s）提升运维效率。

实践笔记

代码风格和错误分析：ESLint + Prettier

  • 安装
# ESlint相关库
pnpm install -D @eslint/js eslint  eslint-plugin-react-hooks eslint-plugin-react-refresh globals

# ts中需要额外安装typescript-eslint
pnpm add -D typescript typescript-eslint

# Prettier 相关库
pnpm install prettier eslint-config-prettier -D

# 提交时自动处理
pnpm install lint-staged -D
  • 初始化eslint
npx eslint --init
  • 配置ESLint
    • 注意：node.js项目中配置，需要去除前端框架(react/vue)相关的插件
// eslint.config.js
import js from '@eslint/js'
import ts from 'typescript-eslint'
import { defineConfig } from 'eslint/config'
import globals from 'globals'
import reactHooks from 'eslint-plugin-react-hooks'
import reactRefresh from 'eslint-plugin-react-refresh'
// 注意 eslint-config-prettier 在 flat config 中的使用方式可能有所不同
// 通常建议使用 plugin:prettier/recommended 的方式集成
import prettier from 'eslint-config-prettier'

export default defineConfig([
  { ignores: ['dist', 'node_modules', 'public'] },
  js.configs.recommended,
  ...ts.configs.recommended, // 使用 configs 来继承推荐配置
  {
    files: ['**/*.{js,jsx,ts,tsx}'],
    languageOptions: {
      ecmaVersion: 'latest', // ecmaVersion 应在此处设置
      globals: {
        ...globals.browser, // 正确合并浏览器全局变量
        // 可以在此添加其他自定义全局变量
      },
      parser: ts.parser, // 明确指定 TypeScript 解析器
      parserOptions: {
        ecmaFeatures: {
          jsx: true, // 只需要 jsx: true, tsx 不是 ecmaFeatures 的有效选项
        },
        sourceType: 'module',
      },
    },
    plugins: {
      'react-hooks': reactHooks,
      'react-refresh': reactRefresh,
    },
    rules: {
      // 建议使用 configs 来继承推荐规则，而不是直接展开 rules
      ...reactHooks.configs.recommended.rules,
      'no-unused-vars': 'off', // 关闭基础规则，使用 TypeScript 版本
      '@typescript-eslint/no-unused-vars': [
        'warn',
        {
          varsIgnorePattern: '^[A-Z_]',
        },
      ],
      'react-refresh/only-export-components': [
        'warn',
        { allowConstantExport: true },
      ],
      // '@typescript-eslint/no-explicit-any': 'warn', // 或 'off'
    },
  },
  // Prettier 集成 - 确保这是最后一个配置以覆盖前面的格式化规则
  prettier,
])
  • eslint检查和修复
    • 检查：pnpm dlx eslint "src/**"，win11系统需要打引号
    • 修复：pnpm dlx eslint --fix
  • 配置Prettier
//prettier.config.js
export default {
    singleQuote: true,
    printWidth: 80,
    trailingComma: 'es5',
    tabWidth: 2,
    semi: false,
    bracketSpacing: true,
    arrowParens: 'always',
    endOfLine: 'lf',
}
  • 忽略相关文件
    • .prettierignore文件来排除不需要格式化的文件/文件夹

# 忽略特定文件类型

    • 终端执行格式化：pnpm prettier --write 。

Git提交规范

commitlint/cli使用

官网：https://commitlint.js.org/

  1. 作用：可以给git commit message进行格式化操作，强制约束提交信息符合规范，这样可以避免杂乱的手写提交信息，提升可读性，也便于版本管理
  2. 步骤：
    • 安装pnpm add @commitlint/cli @commitlint/config-conventional -D
    • .commitlintrc.js配置：添加module.exports = { extends: ['@commitlint/config-conventional'] }

Husky使用

  1. 作用：在git提交时hook阶段处理一些事情，比如代码格式化、类型检查、自动测试等
  2. 步骤：
    • 安装：pnpm add husky -D
    • package.json 配置 husky：scripts属性内添加"prepare": "husky"
    • 初始化：pnpm husk init
    • 新建钩子文件 .husky/commit-msg，用于在提交时检查内容格式
      pnpm dlx commitlint --edit $1
    • 执行git commit -m 'feat: 测试成功案例'：提交时提示成功
    • 执行git commit -m '测试成失败案例'：提交时提示失败
  3. 自动处理错误和格式化
    • 前提：已经配置好eslint和prettier且无错误，并且安装了lint-staged
    • package.json新增
      "lint-staged": {
    "*": "pnpm dlx eslint --fix",
    "*": "pnpm prettier --write"
  },
    • 执行pnpm lint-staged测试是否有错误，需要注意的是lint-staged只会对git暂存区的文件进行自动处理，若无误则继续下一步
    • .husky/pre-commit文件中新增：pnpm lint-staged
    • 执行git add .和git commit -m 'feat: 测试自动处理'：在提交前先执行错误修复和自动格式化代码，无误后进入提交环节，检查提交内容是否符合规范

性能查看及其优化

优化方向和要点

优化维度核心目标关键优化点举例
资源加载减少请求数，降低资源体积文件合并、压缩、图片优化、CDN、缓存策略
渲染效率加速页面解析与呈现关键资源内联、CSS/JS加载顺序优化、减少重排重绘
代码质量提升执行效率代码分割、懒加载、Tree Shaking、高效算法
网络和缓存减少网络延迟与重复传输HTTP/2、Gzip压缩、浏览器缓存、Service Worker

性能查看

Lighthouse

  • 自动化综合审计工具，可生成性能报告并提供优化建议，支持命令行集成到持续集成流程中

Chrome DevTools

  • 适合深入分析
  • 查看方向
    • Network面板：查看所有资源请求的瀑布图，了解加载顺序和耗时
    • Performance面板：记录并分析页面运行时的详细时间线，查看渲染、脚本执行等耗时
    • Performance Insights面板：更直观地查看核心性能指标如LCP（最大内容绘制）的发生过程
