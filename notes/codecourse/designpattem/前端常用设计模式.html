前端常用设计模式

一、简单工厂模式

构建构造器

试想，当我们需要输入很多人的相关信息时，可以很快的写一个构造器保存用户信息，比如下面的代码：

//ES5写法，类
function User(name, age, career){
  this.name = name;
  this.age = age;
  this.career = career;
}

const user = new User(name, age, career)

这样可以在拿到后端接口后直接输入就可以保存了。

  • 那什么是构造器？ 当新建对象的内存被分配后，用来初始化该对象的特殊函数，就叫构造器。
  • 上面的代码其实用的是构造器模式，这个模式中需要辨别哪些是不变的，哪些是变化的，比如上面的代码中，名称、年龄和职业是变化的，但每个人都有上面三个共性属性，这是不变的。

简单工厂模式

上面的代码没有办法区分每个职业之间的区别，我们需要一个字段来描述每个职业是工作内容，这很容易写出如下代码：

function Factory(name, age, career){
  switch(career){
    case "老板":
      return new User(name, age)
    case "财务":
      return new User(name, age)

    //其他职业
  }

  //其他逻辑
}

const user = new Factory(name, age, career)

这样的代码并没有将职业和工作完全区分开，这时思考什么是变化的，什么是不变的。属性是共性的，属性的值是变化的，所以我们有了下面代码：

function User(name, age, career, work){
  this.name = name;
  this.age = age;
  this.career = career;
  this.work = work;
}

function Factory(name, age, career){
  let work = [];
  switch(career){
    case "老板":
      work = ["看报", "喝茶"];
      break;
    case "财务":
      work = ["报表", "发工资"]
      break;

    //...
  }

  return new user(name, age, career, work)
}

const user = new Factory(name, age, career)

这样我们可以无脑传参添加职业和工作内容，不用考虑其他方面的事情。

二、抽象工厂模式（复杂工厂模式）

开放封闭原则

相关信息

1、前端中该原则的使用场景以及实现方式 2、该内容来源于AI总结

开放封闭原则（Open-Closed Principle，OCP）是面向对象设计中的一个重要原则，提出了一种软件设计的理念，即软件实体（类、模块、函数等）应该对扩展开放，对修改封闭。这意味着当需要添加新功能时，应该尽量通过扩展已有的代码来实现，而不是修改已有的代码。 在前端开发中，开放封闭原则的应用可以使代码更加健壮、可维护和可扩展。以下是一些在前端开发中遵循开放封闭原则的常见做法：

  • 使用组件化开发：将页面划分为可重用的组件，每个组件都应该是独立的、高内聚的，对外提供清晰的接口。当需要添加新的功能时，可以通过创建新的组件来扩展现有的功能，而无需修改现有的组件。
  • 采用插件化架构：将功能划分为独立的插件或模块，通过插件的方式来扩展应用程序的功能。新的功能可以通过添加新的插件来实现，而不是修改已有的核心代码。
  • 遵循单一职责原则：确保每个模块、组件或函数只负责一项功能。这样，当需要修改某个功能时，只需修改与之相关的模块，而不会影响其他模块。
  • 利用设计模式：使用设计模式来实现开放封闭原则，例如策略模式、观察者模式等。这些设计模式可以帮助在不修改现有代码的情况下扩展功能。

抽象工厂模式

比如，我们需要开一个门店，需要指定门店的服务人员，可以通过下面的代码实现：

//抽象类：开店
class ETFactory{
  //门店信息
  createStore(){
    throw new Error("抽象类，不允许直接调用，需要重写")
  }

  //服务人员
  createUser(){
    throw new Error("抽象类，不允许直接调用，需要重写")
  }
}

//具体实现类：继承抽象类
class AchieveFactory extends ETFactory{
  createStore(){
    //比如，想去万达开店
    return new WanDanStore()
  }

  createUser(){
    //比如，前台服务
    return new FrontUser()
  }
}


//抽象类：门店
class Store{
  getAdress(){
    throw new Error("抽象类，不允许直接调用，需要重写")
  }
}


//门店 - 具体实现类
class WanDanStore extends Store{
  getAdress(){
    console.log("万达广场xxxxx大厦一楼")
  }
}

//门店 - 具体实现类
class WanXiangStore extends Store{
  getAdress(){
    console.log("万象城xxxxx大厦二楼")
  }
}


//抽象类：服务人员
class User{
  getUser(){
    throw new Error("抽象类，不允许直接调用，需要重写")
  }
}

//服务人员 - 具体实现类
class FrontUser extends User{
  getSkill(){
    console.log("前台接待服务")
  }
}

//服务人员 - 具体实现类
class OtherUser extends User{
  getSkill(){
    console.log("其他服务")
  }
}

//开店：水果店
const fruit = new AchieveFactory();

//门店
const myStore = fruit.createStore()

//服务人员
const myUser = fruit.createUser()

myStore.getAdress()

myUser.getSkill()

当我们想要添加一个门店的时候，不需要去原来的模块修改代码，只需要扩展类就行，比如我们不想开水果店了，想开个饭店，可以这样扩展：

//抽象类：门店
class Restaurant  extends Store{
  createStore(){
    //比如，想去万达开店
    return new WanDanStore()
  }

  createUser(){
    //比如，前台服务
    return new FrontUser()
  }
}

三、单例模式

普通类创建两个实例后，两个实例对象各占一块内存互不干涉，而单例模式要求不论创建多少个实例，都占一块内存，即一个类只能有一个实例。

  • 单例模式不仅能减少不必要的内存开销, 并且在减少全局的函数和变量冲突也具有重要的意义。
  • 有如下两种方法可实现。
  1. 构造函数
class SangleDemo{
  constructor(){
    if(!SangleDemo.instance){
      SangleDemo.instance = this
    }

    return SangleDemo.instance
  }

  someMethod(){
    console.log("待处理的方法")
  }
}

const dome1 = new SangleDemo()
const dome2 = new SangleDemo()
console.log(dome1 === dome2)
  1. 闭包
class SangleDemo{
  someMethod(){
    console.log("待处理的方法")
  }
}

SangleDemo.getInstance = (function(){
  let instance = null
  return function(){
    if(!instance){
      instance = new SangleDemo();
    }
    return instance
  }
})();

const demo1 = SangleDemo.getInstance();
const demo2 = SangleDemo.getInstance();
console.log(demo1 === demo2)
